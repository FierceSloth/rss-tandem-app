# 🏗️ Руководство по созданию UI-компонентов

В нашем проекте используется компонентный подход на базе собственного класса `Component`. Это позволяет нам инкапсулировать логику, стили и разметку, сохраняя код чистым и масштабируемым без использования тяжелых фреймворков.

Данный гайдлайн описывает стандарты разделения логики, правила организации файловой структуры и базовые шаблоны для написания новых компонентов.

---

## 1. Архитектура: Разделение на View и Controller

Каждый сложный компонент должен следовать принципу единой ответственности. Мы строго разделяем визуальное отображение и бизнес-логику.

### 🎨 View (`*.view.ts`)

Отвечает **исключительно** за UI.

- Создает DOM-элементы и выстраивает их иерархию (через метод `append`).
- Применяет классы из SCSS-модулей и управляет ими (например, через `toggleClass`).
- Принимает конфигурацию через `IProps`.
- Слушает базовые события (клики, ввод текста) и передает их наружу, но **не обрабатывает** бизнес-правила внутри себя.

### 🧠 Controller (`*.controller.ts`)

Создается опционально, если компоненту нужна "мозговая" активность.

- Управляет инстансом `View`.
- Делает запросы к API.
- Содержит сложную валидацию, фильтрацию или обработку данных.
- Подписывается на глобальные события приложения.

### Стандартная структура папки компонента:

```text
название-компонента/
├── название-компонента.view.ts       # Отрисовка и структура компонента
├── название-компонента.module.scss   # Локальные стили компонента
└── название-компонента.controller.ts # (Опционально) Бизнес-логика компонента

```

## 2. Категории компонентов: UI, Layout, Features

Для поддержания порядка все компоненты строго делятся на три категории, независимо от того, где они находятся.

1. **`ui` (Базовые элементы)**
   Самые мелкие, "глупые" кирпичики интерфейса. У них почти никогда нет контроллеров. Они получают данные через пропсы и просто их рисуют.
   _Примеры: Button, Input, StatusBadge, Spinner, Checkbox._
2. **`layout` (Каркас и структура)**
   Компоненты, отвечающие за сетку и расположение элементов на экране. Они служат обертками для контента.
   _Примеры: Header, Footer, Sidebar, Card, Modal, Container._
3. **`features` (Бизнес-блоки)**
   Сложные, "умные" компоненты, которые часто имеют свои контроллеры. Они объединяют `ui` и `layout` элементы для выполнения конкретной бизнес-задачи.
   _Примеры: LoginForm, UserProfile, ProductList, ChatWindow._

## 3. Глобальные vs Локальные компоненты

Структура `ui`, `layout`, `features` применяется **на всех уровнях** приложения: как для глобальных компонентов, так и для тех, что используются на конкретной странице.

### 🌍 Глобальные компоненты (`src/components/`)

Сюда попадает всё, что используется более чем на одной странице.

```text
src/
└── components/
    ├── ui/
    │   └── button/
    ├── layout/
    │   └── card/
    └── features/
        └── auth-form/

```

### 🏠 Локальные компоненты страниц (`src/pages/`)

Если компонент специфичен и нужен **только одной странице** (например, уникальный баннер для профиля), он создается в папке этой страницы, но **обязательно сохраняет иерархию** `ui`, `layout`, `features`. Не нужно засорять глобальную папку `src/components/`.

```text
src/
└── pages/
    └── profile-page/
        ├── profile-page.ts
        ├── profile-page.module.scss
        └── components/                  # Локальные компоненты страницы
            ├── ui/
            │   └── profile-avatar/      # Специфичный UI элемент
            ├── layout/
            │   └── profile-grid/        # Сетка, нужная только тут
            └── features/
                └── user-achievements/   # Фича, которая есть только в профиле

```

## 4. Базовый шаблон создания View

Для быстрого создания компонентов в проекте настроен **командный сниппет для VS Code** (находится в `.vscode/components.code-snippets`). Вам не нужно копировать код вручную!

### ⚡ Как использовать автогенерацию:

1. Создайте пустой файл для вашего компонента (например, `user-card.view.ts`).
2. Откройте его, начните вводить слово `view` и нажмите `Tab` (или `Enter`).
3. Редактор автоматически сгенерирует весь базовый код, сам подставит правильное название класса (в PascalCase) и пути до стилей на основе имени вашего файла.

> **Важно:** Если автодополнение не сработало сразу, нажмите `Ctrl + Space` и выберите сниппет с названием `Component View Template` из списка. Убедитесь, что у вас открыта папка проекта как корневая в редакторе.

Сниппет разворачивает следующий шаблон. Он гарантирует правильное наследование от базового класса `Component` и безопасное слияние классов через утилиту `mergeClassNames`:

```typescript
import type { IComponentChild } from '@common/types/types';
import { mergeClassNames } from '@common/utils/class-names';
import { Component } from '@components/base/component';

import styles from './название-компонента.module.scss';

// 1. Описываем интерфейс пропсов (настроек) компонента
interface IProps extends IComponentChild {
  // Добавьте кастомные свойства: color?: string, disabled?: boolean и т.д.
}

// 2. Наследуемся от базового класса Component
export class НазваниеКомпонента extends Component {
  constructor({ className = [] /* дальше ваши кастомные свойства */ }: IProps, ...children: Component[]) {
    // 3. Безопасно сливаем внутренние стили с классами, переданными снаружи
    const cssClasses = mergeClassNames(
      styles.названиеКомпонента, // Укажите ваш главный класс из SCSS модуля
      className
    );

    // 4. Инициализируем родительский класс
    super({ tag: 'div', className: cssClasses }, ...children);

    // 5. Ниже инициализируйте внутренние элементы и вызывайте this.append(...)
    // const chat = new Component({className: styles.chat, ...})
    // this.append(chat)
  }
}

// Используйте данный шаблон и удалите комментарии
```

### Правила работы с классом Component:

1. **Управление разметкой:** Все внутренние элементы должны создаваться как экземпляры `Component` и добавляться в DOM строго через метод `this.append()`.
2. **Внешние классы:** Проп `className` может быть массивом или строкой. Всегда прокидывайте его через `mergeClassNames` вместе со стилями вашего SCSS-модуля.
3. **Удаление:** При уничтожении компонента всегда вызывайте `this.destroy()`, чтобы класс базового компонента очистил слушатели событий, удалил узел из DOM и предотвратил утечки памяти.
